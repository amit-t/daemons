# Playbook: boost (Boost + Borrow)

Boost one engineer's enforceable Local Agent user limit by Borrowing ACUs from donors with the highest projected-safe surplus, keeping total allocation (Σ caps) unchanged unless the user explicitly accepts pool-headroom overage risk. Target email is in Run context (`boost target`). An optional `requested increment` is also in Run context — it is **planning input only, never write authorization**; if `none`, derive the recommendation from the user's run-rate projection (also planning only).

Headroom policy (hard rule 11): the default plan passes `"max_headroom": 250` to `boost_plan_jq`, so the boosted cap never exceeds the recipient's consumed ACUs + 250. A value between 250 and 500 may be passed only when the user explicitly requested it in this session; above 500 is never valid — the jq built-in 500 default is the backstop clamp, which applies even to a requested increment and warns when it clamps.

## Steps

1. **Cycle + roster.** GET `/v3/enterprise/consumption/cycles` for current cycle. GET `/v3/enterprise/members/users` and resolve the target email to `user_id`; build email ↔ user_id map for donor candidates. URL-encode all `user_id` path segments.
2. **Per-user consumption.** Get current-cycle per-user ACUs: one Windsurf `consumption` call (`group_by=user`, cycle start → today) if the key is available, else loop `/v3/enterprise/consumption/daily/users/{user_id}` over the roster. Derive recipient `consumed`, daily `run_rate` (consumed / days elapsed), and `days_left` in the cycle.
3. **Read live current limits.** GET `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` for the recipient and donor pool. Prefer live `local_agent.cycle_acu_limit`; if unset but a fresh ledger has a cap, show both and ask whether to seed missing live limits from the ledger first. The API is authoritative.
4. **Donor candidates — usage patterns, not just the bottom of the table.** Fetch each candidate donor's recent burn to compute `run_rate` (last-7-day average ACUs/day): with the Windsurf key, reuse the single `consumption` call with `granularity=daily` and derive the last-7-day window per user; without it, GET `/v3/enterprise/consumption/daily/users/{user_id}` with `time_after = now − 7d` and divide `total_acus` by 7. Also compute `days_left` in the cycle from the cycle epochs. Donor safety comes first: never cut a donor below `max(consumed + 10% of even share, 50 ACUs)` — raised to projected end-of-cycle consumption + buffer when `run_rate` is known — skip donors with less than 5 ACUs of safe headroom, and do not create 1-ACU skim reductions from low-cap users. With run rates supplied, donors are ranked by highest **projected safe surplus** first (consumed as tie-break): a heavy burner with big nominal surplus is protected, while an idle user with little nominal headroom but weeks of zero usage is a safe donor. Borrow donors give positive ACUs; preview donor `given` as positive and cap movement as `cap_before → cap_after`, not as “negative ACUs.”
5. **Plan.** Build the `boost_plan_jq` input:
   ```json
   {"pool": <DAG_MONTHLY_ACU_POOL>, "share": <pool / N>,
    "recipient_buffer": 0.15, "donor_buffer": 0.10,
    "max_headroom": 250, "days_left": <days left in cycle>,
    "min_donor_cap_after": 50, "min_donor_give": 5,
    "recipient": {"email": ..., "cap": ..., "consumed": ..., "run_rate": ..., "days_left": ...},
    "donors": [{"email": ..., "cap": ..., "consumed": ..., "run_rate": <last-7d ACUs/day>}, ...]}
   ```
   `"max_headroom": 250` is the default per hard rule 11; pass a larger value (500 maximum) only when the user explicitly requested it in this session, never above 500. If Run context has a requested increment, add `"delta_override": <amount>` to `recipient` **as planning input only — it does not bypass the preview or the `CONFIRM DAG WRITE` gate**. Run `jq -f <boost_plan_jq>`. The program returns the recommended cap (`ceil(projected_month_end × 1.15)`, clamped to consumed + `max_headroom` — the clamp also applies to `delta_override` and emits a warning), funded delta, Borrow takes, shortfall, and `sum_before`/`sum_after` (must be equal for zero-sum Borrow). Before preview, reject the plan outright if recipient `cap_after − consumed` exceeds 500 and report that the canonical headroom cap prevents the write. Donor `run_rate` raises that donor's protected floor to `ceil((consumed + run_rate × days_left) × 1.1)`; omit `run_rate` only when recent usage genuinely cannot be fetched.
6. **Preview.** Show a before/after table: recipient Boost (cap_before → cap_after) and every donor Borrow (cap_before → cap_after, positive `given`). Do not show donor changes as negative ACUs. Include donor safe floor when useful, plus `projected_month_end`, `delta`, `funded`, and `sum_before == sum_after`. Surface every warning. If `shortfall > 0`, stop and ask the user to choose exactly one keyword: `PARTIAL` (accept the funded amount only), `CHANGE DONORS` (new named donor scope, replan), `POOL HEADROOM` (run `boost_check_jq`; increases total exposure/overage risk), or `CANCEL` (ends the run). **Never lower donor safety thresholds.** For any non-cancel choice, rerun the named jq planner, show a new complete preview, and require `CONFIRM DAG WRITE`.
7. **Confirm.** Stop after the preview. Write only after the user sends `CONFIRM DAG WRITE` in a later same-session message covering every changed user limit (hard rule 2).
8. **PATCH changed users.** PATCH `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` for the recipient and each donor with body `{"local_agent":{"cycle_acu_limit":<new_cap>}}`. This writes real user-level Local Agent overrides; a user override replaces the default.
9. **Live verify.** GET every changed user limit after PATCH. Confirm `local_agent.cycle_acu_limit` equals the planned value for recipient and donors. On the first failed PATCH or verification GET, quote the exact response body and perform no further PATCH, DELETE, or ledger update in this run; report applied, failed, and unattempted rows, then offer a new read-and-preview run for only that subset.
10. **Update ledger.** Refresh changed entries in the ledger with `{user_id, cap, consumed}` and updated timestamp. Re-read the file and show changed entries. If `shortfall == 0`, `sum_caps` should remain unchanged.
11. **UI instruction.** Print: `app.devin.ai > Enterprise Settings > Consumption` to see current-cycle Local Agent usage by product/user. Note that configured Local Agent limits are API-managed and this run's GET verification is the proof.
