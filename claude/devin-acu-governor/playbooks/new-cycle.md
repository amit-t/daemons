# Playbook: new-cycle (Start-of-cycle full reset)

Reset every Local Agent limit for the new billing cycle and rebuild the whole cap table from scratch: the full monthly pool (`DAG_MONTHLY_ACU_POOL` in Run context, default 24000) is prorated across confirmed active current-member engineers via `compute_caps_jq`. At cycle start consumed ≈ 0, so this collapses to an even split of `pool / N`. Stale explicit overrides on excluded inactive/former users are cleared. The ledger is **rewritten fresh** with the new cycle's epochs — old-cycle data is not merged. Run this once at the start of every billing cycle.

This differs from `dag set-limits` only in intent and guards: `set-limits` re-prorates *remaining* pool mid-cycle; `new-cycle` asserts a fresh cycle first, then rebuilds everything from the full pool.

## Steps

1. **Verify a new cycle is actually current.** GET `/v3/enterprise/consumption/cycles`; current cycle = item with `after <= now < before`. Convert epochs to dates. Then confirm freshness with **either** signal:
   - cycle start (`after`) is within the last ~3 days, **or**
   - current-cycle consumed total is near zero (GET `/v3/enterprise/consumption/daily` for the cycle window; `total_acus` ≲ 1–2% of the pool).
   If **neither** holds, the old cycle is still running: warn loudly — a full reset now would re-grant the full pool mid-cycle and can create real overage — and require an explicit "yes, reset mid-cycle" confirmation before proceeding. Default is to stop and suggest `dag set-limits` instead.
2. **Roster / user IDs.** GET `/v3/enterprise/members/users` (paginate via `end_cursor`). Build current-member rows `{email, user_id, name, member: true}`. URL-encode every `user_id` in endpoint paths (it can contain `|`).
3. **Per-user consumption.** Get current-cycle per-user ACUs (expected ≈ 0 at cycle start, but never assume — measure):
   - Windsurf key available: one GET `consumption` (`product=agent`, `group_by=user`, `start_date`=cycle start, `end_date`=today, `page_size=10000`; follow `next_page_cursor`). Sum `billed_acus` per `user_email`. Mind the 10/hour rate limit.
   - No Windsurf key: loop GET `/v3/enterprise/consumption/daily/users/{user_id}` (URL-encoded) over the roster with the cycle's `time_after`/`time_before`; take each `total_acus`. Batch with `xargs -P4`.
4. **Activity filter — default active members only.** With a Windsurf key, POST `/api/v1/UserPageAnalytics`, join `userTableStats[]` by email, and set `active: true` only when `teamStatus` is active (case-insensitive); missing, inactive, disabled, removed, or former analytics rows default to `active: false`. Without a Windsurf key, activity status cannot be proven: use only current roster members, label active evidence as unavailable in the preview, and ask the user to remove anyone inactive. Do not reserve ACUs for users who are not current members or are inactive.
5. **Confirm engineer set.** Present active headcount N plus active email/user_id/consumed list, then a separate excluded table for inactive or non-current-member users with reasons. Ask the user to confirm engineers included or remove non-engineers/service accounts. Do not PATCH users outside the confirmed active set.
6. **Rebuild the cap table from scratch.** Build `{"pool": <DAG_MONTHLY_ACU_POOL>, "users": [{"user_id": ..., "email": ..., "consumed": ..., "member": true, "active": true|false}, ...]}` and run `jq -f <compute_caps_jq>`. At cycle start (consumed ≈ 0) this yields `team_level` mode: an even `floor(pool / N)` cap per active engineer. With small early-cycle burn it prorates the remainder (`cap_i = floor(consumed_i) + floor((pool − total_consumed) / N)`). Existing overrides are irrelevant to the math — every active engineer gets a freshly computed cap. Surface every warning. The 500-ACU direct-headroom hard rule applies to any per-user manual adjustment layered on top of this even split; prefer ≤ 250.
7. **Read current limits + stale-override cleanup.** For every active target user and every excluded user with a known `user_id`, GET `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` (explicit override only, not inherited default). If an excluded inactive/former user has an explicit `local_agent.cycle_acu_limit`, include a default cleanup action to clear it by PATCHing `{"local_agent":null}` after confirmation. If an excluded non-current member lacks a known `user_id`, list it as not API-addressable and do not write.
8. **Preview and confirm.** Show a table: email, user_id, consumed, old override (or `<unset>`), new `local_agent.cycle_acu_limit`, delta; plus N, pool, `sum_caps` vs pool, the excluded inactive/former table, and stale override cleanup rows. Print UI instruction: `app.devin.ai > Enterprise Settings > Consumption` (usage view; limits are API-managed). Wait for explicit confirmation.
9. **PATCH every active user.** For each confirmed active user, PATCH `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` with body `{"local_agent":{"cycle_acu_limit":<cap>}}`. For each confirmed stale excluded-cap cleanup, PATCH the same endpoint with body `{"local_agent":null}`. Do not include `billing_org_id` unless the user explicitly asked to change attribution.
10. **Live verify.** GET each changed user limit after PATCH. Confirm active users' `local_agent.cycle_acu_limit` equals the planned cap and cleanup users' override is unset. Report applied/failed/verified lists. On any failure, quote the exact body, stop further writes when safe, and offer a resume plan for only the failed subset.
11. **Rewrite the ledger fresh.** Write the ledger file (path in Run context) from scratch with the **new** cycle's `cycle_start`/`cycle_end` epochs, active per-email `{user_id, cap, consumed}`, `sum_caps`, excluded users, cleared excluded-cap users, and the current timestamp. Do **not** merge old-cycle entries — the old cycle's caps and consumption are dead data once the cycle rolls. Create the parent directory if needed. The live API remains authoritative.
12. **Report.** State the cycle window, N, per-user cap, `sum_caps` vs pool, cleared stale overrides, and excluded users. Point to `dag boost <email> [acus]` for mid-cycle top-ups, `dag set-limits-new` for onboarding later hires, and `dag usage` for burn tracking.
