# Playbook: set-limits-new (Seed caps for the uncapped, by Borrowing)

Give an enforceable per-user **Local Agent limit** to active current-member engineers who currently have **no explicit cap** (they ride the inherited org/default limit), and fund those new caps **zero-sum** by Borrowing cap headroom from active current-member users with the **highest safe surplus**. Σ explicit caps for the active eligible set stays unchanged, so the team never tips into overage. This is the targeted complement to `dag set-limits` (which (re)prorates *everyone*): use `set-limits-new` to onboard newly-added active engineers without disturbing existing capped users beyond the borrow.

Default eligible set = current roster members whose activity status is active. Do not seed caps for inactive or former users, and do not count inactive/former capped users as active-member donor headroom. Recipients = active uncapped users. Donors = active capped users, ranked highest **projected** safe surplus first, with lowest consumption as tie-breaker. Recipients are never donors; donors are never re-capped upward. The remaining borrowable budget is prorated **evenly** across recipients (`floor(consumed_i) + min(share, max_headroom)`); the default plan passes `"max_headroom": 250`, a value between 250 and 500 may be passed only when the user explicitly requests it in this session, and above 500 is never valid — the jq built-in default of 500 is the backstop clamp (hard rule 11). When donor headroom cannot fund every recipient, use the `borrow_caps_jq` `recipients_capped` and `recipients_skipped` arrays exactly as returned — do not independently rank recipients; report every skipped recipient unchanged and uncapped (no overage is ever created).

## Steps

1. **Cycle dates.** GET `/v3/enterprise/consumption/cycles`. Current cycle = item with `after <= now < before`. Convert epochs to dates for display.
2. **Roster / user IDs.** GET `/v3/enterprise/members/users` (paginate via `end_cursor`). Build current-member rows `{email, user_id, name, member: true}`. URL-encode every `user_id` in endpoint paths (it can contain `|`).
3. **Per-user consumption.** Get current-cycle per-user ACUs:
   - Windsurf key available: one GET `consumption` (`product=agent`, `group_by=user`, `start_date`=cycle start, `end_date`=today, `page_size=10000`; follow `next_page_cursor`). Sum `billed_acus` per `user_email`. Mind the 10/hour rate limit.
   - No Windsurf key: loop GET `/v3/enterprise/consumption/daily/users/{user_id}` (URL-encoded) over the roster with the cycle's `time_after`/`time_before`; take each `total_acus`. Batch with `xargs -P4`.
4. **Activity filter — default active members only.** With a Windsurf key, POST `/api/v1/UserPageAnalytics`, join `userTableStats[]` by email, and set `active: true` only when `teamStatus` is active (case-insensitive); missing, inactive, disabled, removed, or former analytics rows default to `active: false`. Without a Windsurf key, set `activity_evidence: unavailable` for every current roster member and treat every roster member as provisionally eligible **for preview only**: stop after showing the roster and ask the user to name every exclusion; do not build the final plan or write until the user confirms the final eligible email list (an unmentioned roster member remains included). Do not seed caps for inactive or former users, and do not reserve donor headroom for users who are not current members or are inactive.
5. **Read live current limits — classify capped vs uncapped.** For **every** roster user, GET `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits`. The GET returns an **explicit override only**, not the inherited default. Partition:
   - **Recipients** = active current-member users whose `local_agent.cycle_acu_limit` is **unset** (no explicit override).
   - **Donors** = active current-member users whose `local_agent.cycle_acu_limit` **is** set.
   - **Excluded** = inactive or non-current-member users. They are audited separately and get no new cap reservation by default.
   If a fresh ledger has caps but live is unset, the live API is authoritative — treat as uncapped (recipient) unless the user asks to seed live limits from the ledger first.
6. **Confirm recipient scope — not write authorization.** Present the active uncapped headcount N plus their email/user_id/consumed table, then a separate excluded inactive/former table and capped active donor summary. If activity evidence is unavailable, state that current roster membership is the fallback. Ask the user to reply with the final included recipient emails and exclusions (remove service accounts / non-engineers), and stop for that reply. Scope confirmation does not authorize any PATCH or DELETE — after planning, the separate `CONFIRM DAG WRITE` gate still applies. Do not cap users outside the confirmed active set. If N = 0, report "all active users already have explicit caps — nothing to seed" and stop.
7. **Plan the borrow — usage patterns, not just the bottom of the table.** First fetch each donor's recent burn to compute `run_rate` (last-7-day average ACUs/day): with the Windsurf key, reuse the single `consumption` call from step 3 with `granularity=daily` and derive each donor's last-7-day window; without it, GET `/v3/enterprise/consumption/daily/users/{user_id}` with `time_after = now − 7d` and divide `total_acus` by 7. Compute `days_left` in the cycle from the cycle epochs. Then build the `borrow_caps_jq` input:
   ```json
   {"donor_buffer": 0.10, "max_headroom": 250, "days_left": <days left in cycle>,
    "recipients": [{"email": ..., "user_id": ..., "consumed": ..., "member": true, "active": true|false}, ...],
    "donors":     [{"email": ..., "user_id": ..., "cap": ..., "consumed": ..., "run_rate": <last-7d ACUs/day>, "member": true, "active": true|false}, ...]}
   ```
   `"max_headroom": 250` is the default per hard rule 11; pass a larger value (500 maximum) only when the user explicitly requested it in this session, and never above 500 — the jq built-in 500 default is the backstop.
   Run `jq -f <borrow_caps_jq>`. A donor's `run_rate` raises its protected floor to projected end-of-cycle consumption + buffer (`ceil((consumed + run_rate × days_left) × 1.1)`), so heavy burners with big nominal surplus are protected while idle users with little nominal headroom but weeks of zero usage become safe donors; donors are ranked by highest projected safe surplus first. Omit `run_rate` only when recent usage genuinely cannot be fetched (floor then falls back to consumed + buffer). The program filters out `member:false` and `active:false`, then returns:
   - `mode` — `even_share` (full prorate, `share` ≥ 1), `min_cover` (caps == consumption, no headroom), or `partial` (some recipients could not be funded zero-sum).
   - `recipients_capped[]` `{email, user_id, consumed, cap}` — the new per-user caps.
   - `recipients_skipped[]` — uncapped users left untouched (partial mode).
   - `recipients_excluded[]` / `donors_excluded[]` — inactive or non-current-member rows ignored by default.
   - `donor_takes[]` `{email, cap_before, cap_after, given}` — use this planner output exactly. Borrow draws from highest projected safe surplus first; no donor is cut below its floor: `donor_buffer: 0.10` means a 10-percent buffer as implemented by `borrow_caps_jq` (floor `ceil(consumed × 1.1)`, raised to `ceil((consumed + run_rate × days_left) × 1.1)` when `run_rate` is supplied) — do not add 0.10 ACUs literally or recompute the floor outside jq.
   - `borrowed`, `sum_before`, `sum_after`, `zero_sum` (must be `true`), plus `warnings`.
   Surface every warning. Note: a capped user is never set below their own current consumption (`cap_i ≥ consumed_i`).
8. **Optional stale-cap cleanup.** For each excluded inactive/former user with a known `user_id` and an explicit `local_agent.cycle_acu_limit`, list a proposed cleanup row to clear stale explicit overrides for excluded users with body `{"local_agent":null}`. Do not include a cleanup row in the write set unless the user includes that email in the final preview and later sends `CONFIRM DAG WRITE`. Cleanup lowers total exposure and sits outside the active-set zero-sum calculation. If an excluded non-current member lacks a known `user_id`, list it as not API-addressable and do not write.
9. **Preview and confirm.** Show two tables:
   - **Recipients (new caps):** email, user_id, consumed, new `cycle_acu_limit`, headroom (cap − consumed).
   - **Donors (borrowed from):** email, cap_before → cap_after, given.
   Plus `mode`, `borrowed`, and `sum_before == sum_after` (prove zero-sum for the active eligible set). Include `recipients_excluded`, `donors_excluded`, and any stale override cleanup rows so it is clear inactive/former users receive no new reservation by default. If `mode` is `partial`, do not write: list `recipients_skipped` and ask the user to choose exactly one keyword — `PARTIAL` (keep only the jq-returned capped recipients; still requires a new preview plus `CONFIRM DAG WRITE`), `CHANGE DONORS` (requires a new named donor scope and replanning), or `CANCEL` (ends the run). Pool-headroom changes are out of scope here and require a new command (`dag boost` or `dag set-limits`). Print UI instruction: `app.devin.ai > Enterprise Settings > Consumption`. Then stop; write only after the user sends `CONFIRM DAG WRITE` (hard rule 2).
10. **PATCH changed users.** For each `recipients_capped` user **and** each `donor_takes` donor, PATCH `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` with body `{"local_agent":{"cycle_acu_limit":<cap>}}`. For each confirmed stale excluded-cap cleanup, PATCH the same endpoint with body `{"local_agent":null}`. Do not include `billing_org_id` unless the user explicitly asked to change attribution. Never seed a `recipients_skipped`, `recipients_excluded`, or `donors_excluded` user.
11. **Live verify.** GET each changed user limit after PATCH. Confirm `local_agent.cycle_acu_limit` equals the planned value for every active recipient/donor and is unset for every cleanup user. On the first failed PATCH or verification GET, quote the exact response body and perform no further PATCH, DELETE, or ledger update in this run. Report applied, failed, and unattempted rows, then offer a new read-and-preview run covering only the failed/unattempted subset.
12. **Update ledger.** Refresh the ledger (path in Run context) for every changed active entry with `{user_id, cap, consumed}` and a new timestamp; store excluded inactive/former rows and cleared excluded-cap users separately; re-read and show changed entries. Because the borrow is zero-sum for the eligible set, active `sum_caps` should be unchanged vs before (modulo any FP rounding in the donor buffer), while cleanup rows lower stale inactive/former exposure. The live API remains authoritative; the ledger supports later boost/borrow planning.
13. **Report.** List newly-capped active engineers, donors and what each gave, any active users left uncapped, inactive/former users excluded from reservation, and any stale excluded caps cleared. Point to `dag boost <email> [acus]` for single-user top-ups and `dag set-limits` for a full re-prorate.
