# Playbook: set-limits

Distribute the monthly ACU pool (`DAG_MONTHLY_ACU_POOL` in Run context) across active current-member enterprise engineers as enforceable **per-user Local Agent limits**. The calculation prorates the remaining ACUs after current cycle burn, then PATCHes each target user's `local_agent.cycle_acu_limit` through the Cognizant-provided V3 beta API.

## Targeted mode: `dag set-limits <email>`

If Run context includes `target email`, do **not** run the full-team proration flow. Cap only this target user by using the one-recipient version of the `set-limits-new` Borrow flow:

- Resolve the target email to exactly one active current-member `user_id`.
- Read the target's live `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` setting before planning.
- If the target already has an explicit cap, stop and report that cap; suggest `dag boost <email> [acus]` for raising an existing cap or `dag user <email>` for inspection. Do not borrow or PATCH by default.
- If the target has no explicit cap, build `borrow_caps_jq` input with `recipients` containing **only** the target user and `donors` containing active current-member users with explicit caps. Pass `"max_headroom": 250` by default per hard rule 11 (250–500 only when the user explicitly requests it in this session; above 500 never — the jq built-in 500 default is the backstop). Borrow from active capped donors ranked by highest projected safe surplus (supply donor `run_rate` + `days_left` as in `set-limits-new`). The target is never a donor.
- Do not cap any other uncapped users, even if they appear in the roster. List them only as "left uncapped by targeted mode" if helpful for audit.
- Preview the one target cap plus donor Borrow table. Prove `zero_sum: true`, `sum_before == sum_after`, and that no donor falls below its consumed ACUs + buffer.
- After the user sends `CONFIRM DAG WRITE` (hard rule 2), PATCH only the target and tapped donors, then GET each changed user limit after PATCH and verify exact planned values.

After completing targeted mode, skip the full-team steps below except for shared data-gathering, confirmation, live verification, ledger, and UI-instruction requirements that apply to the target/donor participant set.

## Steps

1. **Cycle dates.** GET `/v3/enterprise/consumption/cycles`. Current cycle = item with `after <= now < before`. Convert epochs to dates for display.
2. **Roster / user IDs.** GET `/v3/enterprise/members/users` (paginate via `end_cursor`). Build current-member rows `{email, user_id, name, member: true}`. This user_id list is required for `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` PATCH targets. URL-encode every `user_id` in endpoint paths.
3. **Per-user consumption.**
   - Windsurf key available: one GET `consumption` with `start_date` = cycle start date, `end_date` = today, `product=agent`, `group_by=user`, `page_size=10000`; follow `next_page_cursor`. Sum `consumption.billed_acus` per `user_email`. One query — mind the 10/hour rate limit.
   - No Windsurf key: loop GET `/v3/enterprise/consumption/daily/users/{user_id}` (URL-encoded) over the roster with the cycle's `time_after`/`time_before`; take each `total_acus`. Batch with `xargs -P4` or similar; no documented rate limit.
   - Cross-check: Σ per-user ACUs should approximate `/v3/enterprise/consumption/daily` `total_acus` for the same window; flag a gap > 5% before any write.
4. **Activity filter — default active members only.** Default eligible set = current roster members whose activity status is active. With a Windsurf key, POST `/api/v1/UserPageAnalytics`, join `userTableStats[]` by email, and set `active: true` only when `teamStatus` is active (case-insensitive); missing, inactive, disabled, removed, or former analytics rows default to `active: false`. Without a Windsurf key, set `activity_evidence: unavailable` for every current roster member and treat every roster member as provisionally eligible **for preview only**: stop after showing the roster and ask the user to name every exclusion; do not build the final plan or write until the user confirms the final eligible email list (an unmentioned roster member remains included). Do not reserve ACUs for users who are not current members or are inactive.
5. **Confirm engineer scope — not write authorization.** Present active headcount N plus active email/user_id/consumed list, then a separate excluded table for inactive or non-current-member users and the reason they are excluded. If activity evidence is unavailable, state that current roster membership is the fallback. Ask the user to reply with the final included engineer emails and exclusions (remove non-engineers/service accounts), and stop for that reply. Scope confirmation does not authorize any PATCH or DELETE — the separate `CONFIRM DAG WRITE` gate still applies after planning. Do not PATCH users outside the confirmed active set.
6. **Compute caps.** Build `{"pool": <DAG_MONTHLY_ACU_POOL>, "users": [{"user_id": ..., "email": ..., "consumed": ..., "member": true, "active": true|false}, ...]}` and run `jq -f <compute_caps_jq>` on it. The program filters out `member:false` and `active:false`, returns per-user `cap` values plus `excluded_users`, and never reserves ACUs for inactive/former users. Mid-cycle formula for eligible users: `cap_i = floor(consumed_i) + floor((pool − eligible_total_consumed) / N)`, so the remaining ACU pool is prorated evenly among confirmed active engineers while preserving what each has already used. Surface every warning for exhausted/near-exhausted pools. Hard rule 11 applies to any per-user manual adjustment layered on top of this proration: 250 ACUs of direct headroom by default, up to 500 only on explicit in-session user request, above 500 never.
7. **Read current limits.** For every active target user and every excluded user with a known `user_id`, GET `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits`. Capture old `local_agent.cycle_acu_limit` (or `<unset>`) and `billing_org_id` if present. Remember: GET returns explicit override only, not inherited default.
8. **Stale excluded-cap cleanup.** If an excluded inactive/former user has an explicit `local_agent.cycle_acu_limit`, include a default cleanup action to clear stale explicit overrides for excluded users by PATCHing `{"local_agent":null}` after confirmation. This removes their individual reservation without inventing a new cap. If an excluded non-current member lacks a known `user_id`, list it as not API-addressable and do not write.
9. **Preview and confirm.** Show a table: email, user_id, consumed, old override, new `local_agent.cycle_acu_limit`, delta, over/under (consumed vs new cap); plus `total_consumed`, `remaining`, `sum_caps` vs pool. Include the excluded inactive/former table, any stale override cleanup rows, and state that excluded users receive no reservation. Print UI instruction: `app.devin.ai > Enterprise Settings > Consumption` (usage view; limits are API-managed). Then stop; write only after the user sends `CONFIRM DAG WRITE` (hard rule 2).
10. **PATCH per-user limits.** For each confirmed active user, PATCH `/v3beta1/enterprise/users/{user_id}/consumption/acu-limits` with body `{"local_agent":{"cycle_acu_limit":<cap>}}`. For each confirmed stale excluded-cap cleanup, PATCH the same endpoint with body `{"local_agent":null}`. Do not include `billing_org_id` unless the user explicitly asked to change attribution.
11. **Live verify.** GET each changed user limit after PATCH. Confirm active users' `local_agent.cycle_acu_limit` equals the planned cap and cleanup users' `local_agent.cycle_acu_limit` is unset. On the first failed PATCH or verification GET, quote the exact response body and perform no further PATCH, DELETE, or ledger update in this run; report applied, failed, and unattempted rows, then offer a new read-and-preview run for only that subset.
12. **Write ledger.** Write the ledger file (path in Run context) with cycle epochs, active per-email `{user_id, cap, consumed}`, `sum_caps`, `excluded_users`, cleared excluded-cap users, and the current timestamp. Create the parent directory if needed. The live API remains authoritative; ledger supports boost/borrow planning.
13. **Report violators / boost candidates.** List active users whose current consumption is already close to or above their new cap, sorted by overage. Point to `dag boost <email> [acus]`, which can Boost one user by Borrowing from low consumers.
14. **Org guardrail reminder.** If the team wants one org-wide hard cap too, use the deterministic one-time command: `dag set limit global <acus> [org_id|org_name]`. Example: `dag set limit global 2400`. It PATCHes the org Local Agent cap and live-GET verifies it.
